# Kong Access Log 诊断分析

根据用户提供的 Kong access log 格式和实际日志，基于四指标五段式方法论进行性能瓶颈定位和问题诊断。

## 输入

`$ARGUMENTS` — 用户提供的日志格式描述和实际日志内容。

如果 `$ARGUMENTS` 为空，提示用户：

> 请提供以下信息：
> 1. **日志格式** — 你 Kong 的 `log_format` 配置（Nginx 格式），或者说明各字段的含义和顺序
> 2. **实际日志** — 粘贴 1-20 条需要分析的实际 access log 行
>
> 示例：
> ```
> 格式：'$remote_addr [$time_local] "$request" status=$status rt=$request_time uct=$upstream_connect_time uht=$upstream_header_time urt=$upstream_response_time'
>
> 日志：
> 10.0.1.1 [07/Jun/2026:10:00:01] "GET /api/orders HTTP/1.1" status=200 rt=3.060 uct=0.003 uht=3.000 urt=3.010
> ```

## 分析步骤

### Step 1 — 解析日志格式

从用户提供的格式中识别出四个核心时间指标的位置和变量名：

| 指标 | 常见变量 | 含义 |
|:----:|:--------|:-----|
| **rt** | `$request_time` | Kong 进程内部总耗时 |
| **uct** | `$upstream_connect_time` | Kong → 上游 TCP 建连（含 DNS + SSL） |
| **uht** | `$upstream_header_time` | 建连 + 发请求 + 等上游首字节 |
| **urt** | `$upstream_response_time` | 建连 + 发请求 + 收完完整响应体 |

同时识别辅助信息：`status`（状态码）、`upstream_addr`（上游地址）、`request`（请求方法和路径）、`time_local`（时间）。

如果用户提供的格式缺少部分指标，明确告知哪些指标缺失以及这对诊断的影响。

### Step 2 — 逐行解析日志

对每条日志行，提取四个核心指标的数值。注意处理 `-1` 和 `-` 两种特殊情况：
- `uct = -1`：请求没有到达上游（Kong 直接拒绝/拦截）
- `uht/urt = -` 或 `-1`：上游未返回响应（连接中断）

### Step 3 — 五段式耗时拆解

对每条有效日志（uct ≠ -1），计算五个时段：

```
rt = ① Kong 前置处理
   + ② uct         （Kong → 上游 网络建连）
   + ③ uht - uct   （上游服务处理）
   + ④ urt - uht   （上游 → Kong 传输响应体）
   + ⑤ rt - urt    （Kong 后置处理 + 发给客户端）
```

正常基线参考：

| 段 | 计算公式 | 正常基线 |
|:--:|:--------|:--------|
| ② 网络建连 | `uct` | < 5ms（同机房）|
| ③ 上游处理 | `uht - uct` | 视业务而定 |
| ④ 传响应体 | `urt - uht` | 通常 < 10ms |
| ①+⑤ Kong 开销 | `rt - urt` | < 10ms |

### Step 4 — 判断口诀

按以下规则对每条日志打标签：

```
uct = -1           --> 🚫 NO_UPSTREAM（请求没到上游，Kong 直接处理）

uct 高              --> 🔴 网络建连问题
                       - 出现 1s/3s/7s 整数秒跳变 → 几乎确定是丢包
                       - 周期性升高 → DNS 缓存过期
                       - 高并发时升高 → TCP 队列满 / keepalive 池耗尽
                       - 所有请求都高 → 跨区域/跨机房

uht - uct 高        --> 🔵 上游服务处理慢
                       - 特定接口高 → 慢 SQL / 锁 / 下游依赖
                       - 全面高 → CPU 打满 / GC 停顿
                       - 首次请求高 → 冷启动

urt - uht 高        --> 🔴/🔵 需二次判断
                       - 小 body 也慢 → 网络带宽/质量问题
                       - 大 body / 特定接口 → 上游流式输出（SSE/chunked）

rt - urt 高         --> 🟡 Kong 自身开销大
                       - 前置慢 → 认证插件（LDAP/OAuth）/ rate-limiting 查库
                       - 后置慢 → logging 插件阻塞 / response-transformer
```

### Step 5 — 结合状态码辅助判断

| 状态码 | uct 值 | 归属 |
|:------:|:------:|:----:|
| 401/403/404/429 | -1 | 🟡 Kong 插件拦截 |
| 502 | 高 | 🔴 网络连接超时 |
| 502 | 正常，uht=- | 🔵 上游崩溃/连接被拒 |
| 503 | 高 | 🔴 DNS 解析失败 |
| 504 | 高 | 🔴 连接上游超时 |
| 504 | 正常 | 🟡 Kong 后置超时 / 🔵 上游处理慢 |

## 输出格式

### 1. 逐行分析表

对每条日志输出一行分析：

```
#N  [status]  标签            connect=  app=      body=     kong=     total=
1   [200]     ✅OK            0.002s    0.045s    0.003s    0.008s    0.058s
2   [200]     🔵APP_SLOW      0.003s    2.500s    0.010s    0.012s    2.525s
3   [502]     🔴NET_CONNECT   0.800s    0.050s    0.005s    0.010s    0.865s
```

### 2. 汇总诊断

- **主要瓶颈：** 标注最突出的层（网络建连 / 上游服务 / 传响应体 / Kong 自身）
- **关键发现：** 列出 2-5 条具体发现（如"uct 出现 1.002s 的整数秒跳变 → 丢包"、"502 全部来自 10.0.1.52 这个 target"）
- **可能根因：** 基于发现推断根因
- **建议排查命令：** 给出 2-3 条针对性的验证命令
- **建议解决方案：** 给出 1-3 条修复建议

### 3. 多条日志时额外输出统计

如果日志超过 3 条，额外输出分位数统计：

```
指标                  Min     Avg     Max     超标比例
网络建连(uct)         0.002s  0.003s  0.008s  0/10
上游处理(uht-uct)     0.040s  0.800s  2.500s  3/10
传body(urt-uht)       0.003s  0.010s  0.050s  0/10
Kong开销(rt-urt)      0.005s  0.012s  0.030s  0/10
```

## 注意事项

- 如果用户日志格式中缺少 `urt`，明确告知：没有 `urt` 无法判断「传响应体」时段，可能遗漏「上游流式输出慢」这类问题
- 如果日志只有 1-2 条，不做统计，只做逐行分析
- 如果日志中出现 uct 的整数秒跳变（1s/3s/7s），重点强调丢包可能性
- 如果多条日志来自不同 upstream_addr，按 upstream 分组分析
- 如果日志中有特定接口路径，提及接口级别的判断（如"仅 /api/orders 慢 → 特定接口问题"）
- 所有时间数值保留 3 位小数（毫秒级精度）
- 使用中文输出分析结果
